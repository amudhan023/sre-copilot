Problem Statement

Why This System Exists

During a P1 incident, an on-call engineer simultaneously correlates metrics, logs, and traces across dozens of services; recalls or searches for relevant runbooks; identifies which deployment or config change triggered the issue; drafts Slack updates while actively debugging; and writes a postmortem days later from memory. This cognitive load at 3am is a systemic organisational risk — not just an inconvenience.

SRE Copilot acts as a first-responder co-pilot. By the time the on-call engineer is paged, the system has already correlated the signals, found the three most similar historical incidents, identified the probable root cause with a confidence score, and drafted the remediation steps. The engineer's job shifts from investigation to validation and execution.

⚡

Autonomous Detection

Statistical z-score gate (2.5σ threshold) + Claude Haiku classification. ~5% of metric events trigger LLM calls. Redis dedup prevents alert storms.

🔗

Temporal Correlation

30-minute Kafka replay reveals the deployment or cascade failure that predated the alert. Impossible with REST APIs that discard history.

📚

RAG-Grounded Reasoning

5 parallel vector searches ground Claude's reasoning in historical incidents, runbooks, architecture docs, and deployment history.

🎯

Calibrated Confidence

Structured confidence score with explicit modifiers. 0.85 = act now. 0.35 = escalate to human. Not a vague "I think it's probably..."

🛡️

Failure-Safe Design

Statistical fallback when LLM is unavailable. Dead letter queue for failed messages. Redis dedup survives partial failures.

♻️

Self-Improving

Each postmortem is indexed into Qdrant. Every resolved incident improves future root cause analysis — the knowledge base grows automatically.

Agent Architecture

The 7 Specialized Agents

Why 7 agents instead of 1? A monolithic agent handling detection, correlation, investigation, remediation, communication, and postmortem would have an unmanageable context window, no parallelism, poor failure isolation, and opaque internal state. Specialization enables each agent to operate at exactly the right model tier, token budget, and latency requirement for its function.

1

Detection Agent

Consumes raw.metrics and raw.logs. Maintains a 50-sample rolling z-score baseline per (service, metric) in Redis. Only events exceeding 2.5σ proceed to LLM classification — this statistical gate filters ~95% of events, keeping token costs low. Claude Haiku classifies the anomaly type and severity with structured JSON output. Redis dedup key (5-min TTL) suppresses alert storms. Publishes AnomalyDetectedEvent to anomalies.detected and inserts incident to Postgres.

Claude Haiku 4.5~500 tokens/event 2.5σ statistical gateRedis dedup 5-min TTL LLM fallback: σ>3.0 auto-escalates
2

Correlation Agent

Consumes anomalies.detected. Uses replay_window() to fetch the last 30 minutes of Kafka telemetry for the affected service and all its known dependencies from the service registry. Detects four correlation signal types: TEMPORAL_PROXIMITY (deployment within 60 min, scored by 1.0 − delta/60), DEPENDENCY_CASCADE (downstream service degradation), RESOURCE_CONTENTION (DB connections > 80), ERROR_AMPLIFICATION (log burst > 3 errors). Runs Claude Sonnet to summarise the incident narrative. Redis active-incident lock (30-min TTL) prevents duplicate incidents for the same service.

Claude Sonnet 4.630-min Kafka replay 4 correlation signal typesBlast radius estimation Active incident dedup lock
3

Investigation Agent

Consumes incidents.opened. Fires 5 parallel Knowledge Retrieval requests via Redis RPUSH/BLPOP. Waits up to 15 seconds per request. Assembles a 4096-token context window from all returned chunks (priority: incidents > runbooks > architecture > deployments > postmortems). Runs Claude Sonnet in tool-use mode with chain-of-thought reasoning and the submit_rca tool. Applies the documented confidence scoring formula to each hypothesis. After publishing rca.completed, starts a background thread polling the triggering metric every 30 seconds to detect recovery.

Claude Sonnet 4.6 tool-use~8000 tokens/incident 5 parallel KR requests4096-token context budget Background resolution monitor
4

Knowledge Retrieval Agent

Synchronous Redis pub/sub server (not a Kafka consumer). Runs a continuous BLPOP loop listening on kr:req:* keys. For each request type, applies the appropriate Qdrant search with hybrid vector + metadata filters. All-MiniLM-L6-v2 embeddings are generated locally (no API key). Cross-encoder reranking (ms-marco-MiniLM-L-6-v2) reranks top-20 → top-5 for INCIDENT_SIMILARITY queries. Token budget is enforced with priority ordering before returning. Pushes response to kr:res:{request_id}.

all-MiniLM-L6-v2 local (384 dims) 5 Qdrant collections Cross-encoder reranking 4096-token budget enforcement
5

Remediation Agent

Consumes rca.completed. Calls KR Agent for a RUNBOOK_LOOKUP on the top hypothesis. Generates an ordered action plan where every step is traceable to a runbook chunk or historical incident resolution — the agent is instructed not to invent steps. Each step includes priority (IMMEDIATE/15MIN/1HOUR), risk_level, rollback, owner, and runbook_source. Queries service registry for on-call contacts for escalation path.

Claude Sonnet 4.6~3000 tokens/plan Runbook-grounded steps Risk-ordered + rollback per step
6

Communication Agent

Subscribes to all 5 lifecycle topics simultaneously. Routes email by role (on-call: everything; leadership: CRITICAL + postmortems; all stakeholders: resolved + postmortem). Renders HTML templates with string substitution. Dispatches via SMTP to Mailhog (local) or real SMTP (production). Logs delivery status to Postgres email_notifications table for audit. Routing uses field-presence heuristics since a single consumer group reads all topics.

5 HTML email templates Role-based routing SMTP via Mailhog Postgres delivery audit log
7

Postmortem Agent

Consumes incidents.resolved. Queries Postgres for the complete incident timeline across all 6 related tables. Runs 6 focused, sequential LLM passes: (1) executive summary — non-technical leadership language, (2) timestamped timeline bullets, (3) primary + contributing root cause, (4) impact quantification with SLA breach analysis, (5) systemic contributing factors, (6) SMART action items with owners and due dates. Each pass has a constrained max_tokens and specific framing. Automatically indexes the postmortem into Qdrant's postmortems collection.

Claude Sonnet 4.6 × 6 passes ~6000 tokens/postmortem Postgres full timeline query Auto-indexes into Qdrant (self-improving)

Resilience Engineering

Edge Cases & Failure Handling

Production systems fail in unexpected ways. Every agent has explicit failure modes and fallback strategies. The goal is graceful degradation — the system continues delivering value even when individual components fail.

Failure ModeImpactMitigation / Fallback
Redis unreachable (startup) All agents retry with 2s delay × 20 attempts (40s total) After 40s, startup fails with clear error. Dedup is skipped if Redis fails mid-operation — alert storm risk accepted over data loss.
Kafka unreachable (startup) All agents retry with 5s delay × 30 attempts (150s total) Kafka is declared ready when list_topics() succeeds. Agent startup is blocked, not crashed — they wait for infrastructure.
Postgres unreachable Incident records cannot be persisted; audit log lost DB operations are wrapped with try/except and log warnings. Agent continues processing events — metrics processing and email delivery still work. Retry pool init every 3s × 20 attempts.
Qdrant unreachable (KR Agent) No knowledge retrieval — RCA proceeds with only correlation signals KR Agent retries connection every 5s × 20 attempts. If searches fail, KRResponse has empty chunks list + error field. Investigation Agent proceeds with reduced context and lower confidence scores.
Kafka consumer partition rebalance Processing pause during rebalance (typically <3s) All agents use enable.auto.commit=True and auto.offset.reset=earliest. Messages processed before rebalance are committed. Messages in-flight during rebalance are reprocessed (idempotency handled by Postgres ON CONFLICT DO NOTHING on incident insert).
Kafka consumer lag (Detection Agent slow) Anomalies detected with delay when metric spike is already resolved Timestamps on all Kafka events are ingestion time, not processing time. Correlation Agent uses the detection_time field for its time windows, not wall clock time. Stale anomalies produce warnings in logs and lower confidence scores.
knowledge-seeder fails mid-run Partial Qdrant population; some collections may be empty Seeder is idempotent: uses content hash to skip existing vectors. On restart: on-failure (Docker Compose), a failed seeder restarts from scratch but only upserts missing vectors. Empty collections result in zero chunks returned from KR Agent (not a crash).
Mailhog unreachable (email delivery) No emails sent; incidents continue to be processed Communication Agent logs SMTP failures and records status=FAILED in Postgres. The agent does not retry (email idempotency is hard to guarantee). In production, a real SMTP provider with retry semantics would be used.
Failure ModeImpactMitigation / Fallback
Anthropic API rate limit LLM call delayed Exponential backoff: waits 60s × attempt_number. Up to 3 retries for RateLimitError. During wait, the Kafka consumer is blocked — messages queue but are not lost.
Anthropic API error (5xx) LLM call fails 3 retries with 5s × attempt_number backoff. After 3 failures: Detection Agent falls back to statistical-only severity determination; Investigation Agent proceeds with reduced context and explicitly low confidence (0.30).
LLM returns malformed JSON Hypothesis parsing fails extract_json_block() extracts JSON from markdown code fences or bare JSON. If parsing still fails, a fallback hypothesis with "investigation inconclusive" is generated — the incident is not dropped.
LLM tool-use max rounds exceeded (Investigation) RCA incomplete — agent loop terminates early Max 12 rounds. If reached without submit_rca call, falls back to a direct JSON extraction prompt (non-tool-use). Returns a low-confidence hypothesis with the raw response.
LLM hallucinates runbook steps (Remediation) Action steps may be incorrect Agent is instructed to cite runbook_source per step. Steps without citation are labeled "SRE best practice" — the on-call engineer can recognise ungrounded steps. The email template renders the source citation prominently.
LLM prompt injection via log content Malicious log lines could attempt to override agent behavior Log content is never concatenated into system prompts. External data is always passed as structured tool result content. Agent outputs are validated against Pydantic schemas before Kafka publication. Log messages are length-truncated to 500 chars before LLM consumption.
Postmortem LLM pass returns empty text One section of the postmortem is blank Each of the 6 passes is wrapped in try/except. An empty result is stored as "" for that section. The postmortem is assembled from all passes — a missing section is visible in the output, prompting human review (which is always recommended anyway).
Failure ModeImpactMitigation / Fallback
Anomaly detected before baseline established (<10 samples) High false positive rate during warmup period Hard minimum of 10 samples (MIN_SAMPLES) before z-score is computed. During warmup, metric events are buffered to Redis baseline but never trigger anomaly detection. ~2 minutes to establish baseline at 15s scrape interval.
Alert storm: 100+ anomalies/minute LLM costs spike; downstream agents flooded Redis dedup key per (service, anomaly_type, metric) with 5-min TTL limits each combination to 1 anomaly per 5 minutes. Active incident lock per service (30 min) prevents multiple parallel incidents per service. Practical limit: ~6 unique anomalies/5 min × number of services.
Metric baseline permanently shifted (new normal) False positives after a scaling event or config change Rolling window of 50 samples adapts to new baseline over ~8 minutes (50 × 10s publish interval). During adaptation, sigma threshold still requires significant deviation from the mixed old/new baseline. Production mitigation: increase window to 100+ samples, add seasonal decomposition.
Kafka replay returns no messages (empty window) Correlation Agent has no context for correlation Correlation agent proceeds with zero signals and no deployment context. Publishes incidents.opened with empty correlation_signals list. Investigation Agent still runs but confidence will be lower (no TEMPORAL_PROXIMITY or CASCADE signals).
Qdrant returns no similar incidents (novel anomaly type) No historical context for RCA KRResponse has empty chunks for INCIDENT_SIMILARITY. Confidence formula applies the −0.15 no similar incidents modifier. Investigation Agent still reasons from correlation signals and runbooks. Outcome: lower confidence score, higher likelihood of human escalation recommendation.
Deployment event timestamp skew (>60 min delay in indexing) Missed deployment correlation Deployment ingester uses a Postgres cursor; delayed indexing means the event arrives late to the raw.deployments topic. The Correlation Agent's deployment window is configurable (default 60 min). In production, a webhook receiver would eliminate this delay.
Postmortem Postgres query returns empty timeline Postmortem generated with minimal context Postmortem Agent queries for agent events and related tables, but falls back to the incident record itself if the timeline is empty. A minimal postmortem with at least the detection time, RCA result, and MTTR is always generated.
Edge CaseScenarioHow It's Handled
Cascade failure creates N incidents api-gateway degrades → payment-service degrades → order-service degrades. Each triggers a separate Detection Agent event. Active incident Redis lock per service. Only the first service to be detected opens an incident. Subsequent services are either: (a) identified as cascade by the Correlation Agent in the first incident, or (b) suppressed when their own anomaly is checked and an active incident lock exists for a related service.
Incident resolves before RCA completes Anomaly spikes and recovers in 90 seconds — faster than the RCA pipeline. Resolution monitor in Investigation Agent polls every 30 seconds. If recovery is detected while RCA is still in progress, the resolution event is queued. RCA completes first (publishing rca.completed), then incidents.resolved is published. Postmortem still generated. MTTR reflects actual duration.
Multiple deployments in the correlation window 3 services deployed in the last 60 minutes when an incident occurs. Correlation Agent fetches all deployments in the window and picks the most recent one (sorted by deployed_at DESC). The most recent is the most likely cause. Additional deployments are noted in the correlation context but don't create separate hypotheses.
Confidence score ties between hypotheses Two hypotheses have identical confidence score after modifiers. Hypotheses are sorted by confidence descending, then by rank from the LLM response. The competing_hypothesis_count > 2 modifier (−0.10) applies to both, reducing overall confidence and signalling ambiguity to the on-call engineer.
Same anomaly re-fires after dedup TTL expires A persistent issue (e.g., slow memory leak) triggers a new detection every 5 minutes after the dedup key expires. The active incident lock (30-min TTL) prevents a new incident from being opened while the first is still in progress. After 30 minutes, if the issue is still ongoing, a new incident is created — which is the correct behaviour (the first incident may be stale or may have been incorrectly auto-resolved).
Postmortem auto-indexed into Qdrant produces incorrect future recalls A low-quality postmortem (e.g., LLM wrote generic content) pollutes the knowledge base. Postmortems are indexed with all metadata including the incident_id. The postmortem email contains an explicit "Human review required before sharing externally" disclaimer. Future RAG over postmortems is lower-priority (postmortems < incidents < runbooks in the token budget), reducing blast radius of a bad postmortem.

Architecture Reasoning

Design Decisions & Tradeoffs

Each architectural choice has an explicit rationale and acknowledged tradeoff. These decisions reflect the constraints of a local demo that must also be extensible to production.

Kafka over RabbitMQ / Redis Streams

✓ Chosen: Apache Kafka

Log-based storage enables event replay. The Correlation Agent fetches the 30 minutes of telemetry that predated the incident — impossible with message-queue semantics that delete on consumption. Also: ordered partitioned streams enable temporal correlation; consumer groups support independent agent scaling; 14-day retention creates an audit trail without additional infrastructure.

✗ Considered: RabbitMQ / Redis Streams

RabbitMQ deletes messages on consumption — replay requires a separate time-series store. Redis Streams is lighter but lacks the ecosystem maturity and partition-based parallelism. The replay requirement alone makes Kafka the only correct choice.

Tradeoff accepted: Kafka's operational complexity (Zookeeper in demo mode, broker management in production) vs. the replay capability that enables temporal correlation. For a demo, the Docker Compose setup hides this complexity entirely.

Qdrant over pgvector (Postgres extension)

✓ Chosen: Qdrant

Hybrid filtering: combined vector similarity + metadata predicates in one query (service_name=payment-service AND resolved=true AND environment=production). HNSW index tuning. Zero schema migrations for new payload fields. Local Docker-first with full API — no cloud dependency.

✗ Considered: pgvector

Keeps vector data co-located with relational data (one less service). But lacks the rich filtering API, has less mature approximate search, and adds cognitive load to the Postgres schema. For a demo, the extra service cost is negligible; for production, Qdrant's dedicated performance is worth it.

Tradeoff accepted: One more Docker service to manage vs. significantly better retrieval quality and filtering power. pgvector would work for <50k vectors with simple queries; Qdrant scales to millions.

Synchronous Redis pub/sub for Knowledge Retrieval (not Kafka)

✓ Chosen: Redis RPUSH/BLPOP

The Investigation Agent's LLM reasoning needs knowledge before it can reason. Async Kafka patterns would require the LLM to handle partial context or wait indefinitely. Synchronous BLPOP with 15-second timeout gives a clear success/failure contract and enables the Investigation Agent to fire 5 parallel requests and wait for all 5 before proceeding.

✗ Considered: Kafka request/response

Kafka supports request/response via correlation IDs, but the pattern is complex to implement and adds latency from broker round-trips. For a synchronous, low-latency, high-reliability communication pattern within a single Docker network, Redis BLPOP is the correct choice.

Tradeoff accepted: KR Agent becomes a synchronous bottleneck rather than a scalable async consumer. Mitigation: multiple KR Agent replicas could serve from the same Redis key space. For the demo, one instance is sufficient.

Claude Haiku for Detection, Sonnet for Investigation

✓ Chosen: Model Tiering

Detection runs on every metric event — high volume, simple task (classify anomaly type from numeric values). Haiku is 5× faster and 20× cheaper. Investigation is a complex multi-step reasoning task with long context windows requiring strong reasoning. Sonnet is the right model for this.

✗ Considered: Sonnet everywhere / Haiku everywhere

Sonnet everywhere: 20× higher cost, unnecessary quality for classification. Haiku everywhere: insufficient reasoning quality for multi-step root cause analysis with tool use — hypotheses would be shallow.

Tradeoff accepted: Slightly more complex deployment (two model IDs to manage) vs. 10× cost reduction and correct quality at each stage.

all-MiniLM-L6-v2 (local) over text-embedding-3-large (OpenAI)

✓ Chosen: Local all-MiniLM-L6-v2

Zero additional API key required. Runs in Docker with no external call. 384 dimensions is sufficient for domain-specific retrieval over operational incidents (all vocabulary is similar — "connection pool", "latency spike", "deployment", "rollback"). Seeder runs offline in the container.

✗ Not default: text-embedding-3-large

3072 dimensions, higher recall for diverse domains. But requires an OpenAI API key and per-call cost (even at $0.00013/1K tokens, seeding 280 vectors = $0.04). Accessible via EMBEDDING_MODEL=text-embedding-3-large env var.

Tradeoff accepted: Slightly lower retrieval quality (detectable primarily for novel anomaly types not seen in training data) vs. zero additional dependencies for demo. Upgrade path is a single env var change.

7 Agents over 1 Monolithic Agent

✓ Chosen: 7 Specialized Agents

Parallel execution (Correlation and KR Agent run concurrently). Independent scaling (Detection processes every event; Postmortem fires rarely). Failure isolation (bug in Remediation doesn't break Detection). Observability (every agent decision is a discrete Kafka event). Correct model tier per agent (Haiku for detection, Sonnet for reasoning).

✗ Considered: Monolithic Agent

Single agent with 100K+ token context window attempting detection + correlation + investigation + remediation + communication + postmortem. Context window saturation causes quality degradation. Cannot run detection at high throughput while also running slow postmortem generation. Single point of failure.

Tradeoff accepted: Kafka coordination overhead between agents vs. clean separation of concerns. The overhead is negligible (<5ms per Kafka publish) compared to the LLM call latency.

Future Development

Roadmap & Upgrade Paths

The current system is a functionally complete demo. These are the upgrades that would be implemented for a production deployment at scale, listed by priority and complexity.

Phase 2 — Production Hardening

Slack & PagerDuty Integration

  • Real-time Slack thread creation in the service channel (#payments-alerts)
  • PagerDuty page with enriched incident context (not just an alert)
  • Slack button to acknowledge, resolve, or request human investigation
Phase 2

Human-in-the-Loop Approval

  • For HIGH risk remediation steps: require on-call approval before sending
  • Slack interactive buttons: "Approve" / "Skip" / "Modify"
  • Timeout: auto-send after 5 minutes if no response (configurable)
Phase 2

SLA Tracking & Breach Detection

  • Compare observed P99 against service registry SLA targets
  • Compute SLA breach duration per incident
  • Escalation policy triggered at breach: page service owner, then manager
Phase 2

Auto-Remediation (Low-Risk Steps)

  • Execute IMMEDIATE + LOW risk steps automatically (e.g., scale replicas)
  • Kubernetes API integration for horizontal scaling
  • Rollback trigger via CI/CD webhook for DEPLOYMENT-correlated incidents
  • Gate: confidence > 0.80 AND risk_level = LOW required
Phase 3 — Advanced AI

Predictive Detection

  • Prophet/LSTM time series forecasting per metric
  • Alert when predicted value will exceed threshold in next 15 minutes
  • Seasonal decomposition to handle diurnal traffic patterns
  • Reduces MTTR from "after onset" to "before impact"
Phase 3

Knowledge Graph (Neo4j)

  • Service dependency graph stored in Neo4j with traversal queries
  • Multi-hop blast radius: "what breaks if payment-service is slow?"
  • Historical causality chains: link recurring incidents to systemic causes
  • Replaces the current static service_registry table
Phase 3

Distributed Tracing Integration

  • Jaeger/Tempo integration for trace correlation
  • Correlation Agent can identify the specific span and service hop that introduced latency
  • RCA precision improvement: from "payment-service is slow" to "specific DB query in PaymentProcessor.validate() is slow"
Phase 3

Multi-Model Ensemble

  • Run Investigation Agent on both Claude Sonnet and a local open-source model (Llama 3)
  • Merge hypotheses: include in top candidates only if both models agree
  • Improves confidence calibration; reduces single-model hallucination risk
  • Cost optimisation: use open-source model for low-risk classifications
Phase 4 — Ops Excellence

Feedback Loop & Fine-Tuning

  • On-call engineer rates RCA accuracy after each incident (thumbs up/down)
  • Negative feedback adjusts embedding weights for similar incident retrieval
  • Monthly fine-tuning run on the org's specific incident taxonomy
  • Confidence calibration drift detection: alert if confidence vs. accuracy diverges
Phase 4

Chaos Engineering Integration

  • Pre-defined chaos experiments that test agent detection coverage
  • Scheduled weekly chaos runs during low-traffic periods
  • Metric: % of chaos scenarios detected within 60 seconds
  • Regression gate: chaos run in CI before deploying agent changes
Phase 4

Multi-Cloud & Multi-Stack Support

  • CloudWatch metrics ingester (AWS)
  • Azure Monitor log ingester
  • GCP Cloud Monitoring support
  • Ingestion layer is already decoupled — adding a new source = new ingester service only
Phase 4

MTTR Analytics Dashboard

  • Historical MTTR trend per service and anomaly type
  • False positive rate over time (confidence score vs. actual accuracy)
  • Knowledge base freshness metric (when were runbooks last updated?)
  • Agent latency percentiles per investigation phase

Quick wins that could be done today: (1) Add Prometheus alert_rules.yml firing to raw.alerts Kafka topic via alertmanager webhook — this adds a pass-through detection path that requires no LLM call. (2) Enable text-embedding-3-large via EMBEDDING_MODEL env var for better retrieval quality. (3) Increase SIGMA_THRESHOLD to 3.0 in .env to reduce false positives for stable environments.
