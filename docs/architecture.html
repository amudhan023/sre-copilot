High-Level System Architecture

SRE Copilot is organized into five horizontal layers. Data flows top-to-bottom from simulated production traffic through ingestion, event routing, AI processing, and finally into the application and monitoring layer. The Knowledge Layer sits alongside the Agent Layer providing a RAG (Retrieval-Augmented Generation) service that agents query synchronously.

How failure injection works: The failure-injector writes anomalous metric values to Redis at key failure:state:{service}. The traffic-simulator reads this state on every metric generation tick and blends it into the Prometheus metrics endpoint. This decouples failure definition from metric generation — adding a new scenario requires zero changes to the traffic simulator.

Section 3

Kafka Event Bus Design

All inter-agent communication flows through Kafka (except the synchronous KR pub/sub). Using an event log instead of REST is a deliberate architectural choice — it enables temporal replay (Correlation Agent reads 30 minutes of history), guaranteed delivery during agent restarts, and a first-class audit trail.

Figure 3.1 — Kafka topic topology showing 5 topic groups, consumer group assignments, and message flow direction. 14 topics total across 5 semantic groups.
Why Kafka enables temporal correlation: The Correlation Agent's killer feature is replay_window() — it seeks the raw.metrics consumer to a timestamp 30 minutes in the past and reads forward, collecting metric history for the affected service and its dependencies. This pattern is impossible with REST APIs that don't retain history. The entire correlation context is assembled from a single Kafka log replay with no additional storage required.

Agent Pipeline Overview

Seven autonomous agents form a sequential pipeline triggered by Kafka events. Each agent owns exactly one concern, uses the minimum model capability required for its task, and communicates exclusively via Kafka (except Knowledge Retrieval which uses synchronous Redis pub/sub).

Figure 4.1 — Full agent pipeline. Agents 1–3 and 5–7 form a Kafka-driven pipeline. Agent 4 (Knowledge Retrieval) is a synchronous service using Redis pub/sub, called by Investigation and Remediation agents. The resolution monitor is a background thread inside the Investigation Agent.
Section 5

Detection Agent — Deep Dive

The Detection Agent is the system's first line of defense. It uses a two-tier approach: a cheap statistical gate that runs on every event, followed by an LLM classifier that only fires when the statistical gate passes. This ensures the LLM is never called on noise.

Figure 5.1 — Detection Agent internal flow. Only ~5% of metric events exceed the 2.5σ threshold and proceed to LLM classification. Redis dedup prevents re-processing the same anomaly within the 5-minute TTL window.

Z-Score Algorithm

def compute_zscore(values: list[float], new_value: float) -> tuple[float, float, float]: # Returns (zscore, mean, std). Returns (0, new_value, 0) if insufficient samples. if len(values) < MIN_SAMPLES: # MIN_SAMPLES = 10 return 0.0, new_value, 0.0 mean = sum(values) / len(values) variance = sum((v - mean) ** 2 for v in values) / len(values) std = math.sqrt(variance) if variance > 0 else 0.0 if std < 1e-9: # constant metric — never anomalous return 0.0, mean, std return (new_value - mean) / std, mean, std # Rolling baseline: LPUSH → LTRIM to last 50 values (BASELINE_WINDOW) # Key: metric:{service}:{metric_name} # Current value excluded from baseline to avoid self-comparison bias zscore, mean, std = compute_zscore(history[1:], value) # history[0] is current value

Log Anomaly Detection (Secondary Path)

In addition to metric-based detection, the Detection Agent monitors raw.logs for error bursts. If a service emits ≥5 ERROR/FATAL log entries within 60 seconds (tracked via INCR error_count:{service} with 60s TTL in Redis), it fires an ERROR_RATE_SPIKE anomaly with severity HIGH/CRITICAL. This handles cases where the error rate metric hasn't yet been scraped but the logs are already showing failures.

Section 6

Correlation Agent — Deep Dive

The Correlation Agent converts an isolated anomaly signal into a rich incident context. It does this by replaying 30 minutes of Kafka history, querying the deployment database, checking downstream services for cascade effects, and assembling four types of correlation signals.

Figure 6.1 — Correlation Agent flow. All four data collection operations run in parallel (Python threads). The Redis active_incident lock prevents duplicate incidents for the same service within a 30-minute window, which is how cascade failures are grouped under a single incident ID.
Section 7

Investigation Agent — Parallel KR + Confidence Scoring

The Investigation Agent is the system's primary reasoning engine. It orchestrates 5 parallel Knowledge Retrieval requests, assembles a rich context document, runs a tool-use LLM loop for chain-of-thought RCA, then scores each hypothesis using a deterministic formula.

Figure 7.1 — Investigation Agent flow. 5 KR requests fire in parallel via ThreadPoolExecutor, results are assembled with token budget enforcement, then Claude Sonnet runs a tool-use loop that calls get_incident_context and submit_rca to finalize the hypothesis list.

Confidence Scoring Formula

Confidence is computed deterministically from observable evidence — not from LLM natural-language claims. This allows the on-call engineer to see exactly why the system is or isn't confident.

base = 0.50 # epistemic prior: unknown cause +0.20 if deployment_correlated +0.15 if len(similar_incidents) > 0 +0.10 if len(similar_incidents) >= 3 # strong historical signal +0.05 if has_runbook +0.05 if len(evidence) >= 4 −0.15 if not similar_incidents −0.10 if competing_hypothesis_count > 2 # ambiguous signal final = clamp(base + modifiers, 0.0, 1.0) # Confidence interpretation: # ≥ 0.80 → rollback deployment immediately # 0.60–0.79 → follow runbook, monitor closely # 0.40–0.59 → investigate manually alongside AI suggestion # ≤ 0.39 → insufficient evidence — escalate to senior engineer
Section 8

Knowledge Base Architecture (RAG Layer)

The RAG layer uses Qdrant with 5 purpose-built collections. The design is non-obvious: incidents are chunked into 4 semantic types per incident, enabling precise retrieval of symptoms independent of root cause, which is the pattern you want when a new incident presents similar symptoms to a historically different root cause.

Figure 8.1 — Five Qdrant collections with their schema, chunk counts, filter predicates, and token budget priority order. The incidents collection splits each historical incident into 4 semantic chunk types, enabling precise symptom-matching independent of root cause.

Why Not pgvector?

Qdrant advantages for this use case

  • Hybrid filtering — Combined vector similarity + metadata predicates in a single query (e.g., env=production AND resolved=true AND service_name=payment-service). pgvector requires post-filter which wastes the ANN index.
  • HNSW tuning — m=16, ef_construct=100 directly controls recall/latency tradeoff. pgvector exposes fewer knobs.
  • Schema-free payloads — Adding new metadata fields to any collection requires no ALTER TABLE, just write the new key in the payload dict.
  • Docker-native — Full-featured local instance with zero configuration. Not a cloud dependency.

Embedding model choice

all-MiniLM-L6-v2 (384-dim, local) vs text-embedding-3-large (3072-dim, OpenAI API):

  • MiniLM runs locally — no additional API key, zero per-call cost, no network hop
  • Sufficient for domain-specific operational text (incidents all share vocabulary and structure)
  • 384-dim HNSW index is 8× smaller than 3072-dim — faster ANN lookup
  • To switch: set EMBEDDING_MODEL=text-embedding-3-large in .env

Section 11

Postmortem Agent — 6-Pass LLM Generation

Each postmortem section has different constraints (length, audience, format). A single unconstrained prompt produces mediocre output across all sections. Six focused passes each with a specific directive produce expert-level output per section.

Figure 11.1 — 6-pass postmortem generation. The same incident context is prepended to all 6 prompts, each with a focused directive. Generated postmortems are indexed into Qdrant, making every resolved incident part of the permanent knowledge base for future investigations.

Cost Optimisation Mechanisms (Already Implemented)

Model Tiering

Detection uses claude-haiku-4-5 (~8× cheaper than Sonnet). Only agents that need multi-step reasoning use Sonnet. KR uses local embeddings — zero API cost.

Statistical Gating

Only ~5% of raw metric events exceed the 2.5σ threshold and proceed to LLM classification. The z-score gate eliminates noise before any tokens are spent.

Redis Deduplication

5-minute TTL dedup prevents re-processing identical anomalies. 30-minute active_incident lock prevents duplicate incidents. Both eliminate redundant LLM calls.

Token Budget Enforcement

KR Agent caps context at 4,096 tokens (prioritized by source type) before returning to Investigation Agent. Prevents runaway context windows in the RCA loop.

max_tokens Caps

Every LLM call specifies explicit max_tokens: Detection=200, Correlation=300, Investigation=4096, Remediation=2048, Postmortem passes=300–800 each.

Focused Pass Design

Postmortem 6-pass approach uses small, focused prompts rather than one large unconstrained prompt. Each pass has a strict output constraint — 2-3 sentences, ≤10 bullets, etc.

Shared Library (shared/)

All 7 agents and all 3 ingesters import from a single shared library at /app/shared/ (mounted into each Docker container). This ensures consistent serialization, retry logic, and client patterns.

models.py — 20 Pydantic Event Models

  • RawMetricEvent, RawLogEvent, RawDeploymentEvent — telemetry events
  • AnomalyDetectedEvent — detection output with sigma, p50/p95/p99
  • CorrelationSignal, IncidentOpenedEvent — correlation output
  • RootCauseCandidate, DeploymentCorrelation, RCACompletedEvent
  • RemediationStep, RemediationPlanEvent
  • IncidentResolvedEvent, PostmortemGeneratedEvent, PostmortemDocument
  • KRRequest, KRResponse, KnowledgeChunk — KR pub/sub models
  • ServiceInfo — service registry model
  • Enums: EventType, Severity, AnomalyType, IncidentStatus, KRQueryType

kafka_client.py — Producer/Consumer/Replay

  • make_producer() — confluent-kafka producer with linger_ms=10
  • make_consumer(topics, group_id, auto_offset_reset)
  • publish(producer, topic, data_dict, key) — JSON serialize + produce
  • consume_loop(consumer, handler) — poll loop with error handling
  • replay_window(topic, since_ms, until_ms, service_filter, max_messages) — temporal seek

llm_client.py — Claude API Wrapper

  • chat(system, user, model, max_tokens, temperature) — single-turn with 3× retry + exponential backoff
  • run_tool_use_agent(system, prompt, tools, tool_executor, max_rounds=12) — full agentic tool-use loop
  • extract_json_block(text) — robust JSON extraction from markdown or bare JSON

redis_client.py — All Redis Patterns

  • push_metric(service, metric, value, max_len=50) — LPUSH+LTRIM
  • get_metric_history(service, metric, count) — LRANGE
  • set_dedup(key, ttl) — SET NX with TTL
  • get/set_active_incident(service, incident_id, ttl)
  • delete_active_incident(service) — on resolution
  • push/pop_kr_request(request_id, payload) — BLPUSH/BLPOP
  • push/wait_kr_response(request_id, timeout)
  • get/set_failure_state(service)
  • incr(key, ttl) — error burst counter

db_client.py — PostgreSQL CRUD

  • init_pool() — psycopg2 ThreadedConnectionPool (1–10 connections)
  • insert_incident(id, data) — initial detection insert
  • update_incident(id, updates) — status + JSONB column updates
  • get_incident(id) — full row fetch
  • log_agent_event(incident_id, agent, event_type, payload)
  • get_incident_timeline(incident_id) — ordered agent_events
  • get_service_info(name), get_service_dependencies(name)
  • get_recent_deployments(services, since_ms, window)
  • insert_postmortem(...), log_email(...)
